The Requirements Hierarchy

Requirements exist at different levels of abstraction. The BA's job is to work from the most abstract (what the business needs to achieve) down to the most concrete (what specific functions and data the system must handle), while ensuring each level is traceable to the one above it.

Business

Business Requirements

The high-level outcomes the organisation needs the project to produce. "Reduce export documentation processing time by 40%." No system, no solution — just the goal.

Stakeholder

Stakeholder Requirements

What specific people need to be true about the system from their perspective. "The merchandiser needs to generate a complete commercial invoice from a PO in under three minutes." Tied to a person and a context.

Functional

Functional Requirements

What the system must do — specific, testable functions. "The system shall allow a user with the 'Merchandiser' role to generate a commercial invoice from a selected PO using data from the PO header, line items, and buyer master record."

Non-functional

Non-functional Requirements

Quality attributes the system must exhibit. "The invoice generation function shall complete within 5 seconds for POs with up to 100 line items." Defines how the system performs, not what it does.

Every functional requirement should trace back to at least one stakeholder requirement, which traces back to at least one business requirement. If you cannot draw that line, question whether the requirement belongs in scope at all.

Functional Requirements: Writing the SRS Section by Section

A Software Requirements Specification (SRS) is the formal document that captures functional and non-functional requirements in sufficient detail for development to begin. For most projects in Bangladesh's SME sector, a full IEEE-standard SRS is overkill. A "SRS-lite" — structured but streamlined — achieves the same result with a fraction of the bureaucratic overhead.

Principles for writing good functional requirements:

  • Use "shall" for mandatory requirements — "The system shall…" is unambiguous. "The system should be able to…" leaves room for negotiation that does not belong in a spec.
  • One requirement per sentence — If you use "and" in a functional requirement, split it into two. Each requirement should be independently verifiable.
  • Name the actor — "The system shall allow a user to…" is weaker than "The system shall allow a user with the Merchandiser role to…"
  • Specify the trigger and the outcome — "When the user clicks Export, the system shall generate a PDF of the commercial invoice containing [fields]." Trigger → action → output.
  • Include error handling — "If the PO is in a Cancelled status, the system shall prevent invoice generation and display the message: 'Invoice cannot be generated for cancelled orders.'"

RMG Example

The factory MD's requirement: "I want full visibility into our production orders." This cannot be a functional requirement — it is a business aspiration. A good BA unpacks it:

Functional requirements derived from "full visibility":
FR-01: The system shall display all open production orders on a single screen, filterable by buyer, product type, and production status.
FR-02: Each production order record shall include planned start date, actual start date, planned completion date, and current completion percentage.
FR-03: The system shall highlight any production order where completion percentage is more than 10% below the planned percentage for the current date.
FR-04: The system shall allow the MD role to export the production order view as an Excel file.

"Full visibility" is now four testable requirements. A developer can build them. A tester can verify them.

Data Requirements & Data Dictionary

Data requirements define what information the system stores, how it is structured, and what the rules governing it are. This is where most BAs underinvest — and where developers spend the most time making assumptions that later cause migration problems or integration failures.

For each data entity, document:

  • Field name — exact name as it should appear in the system
  • Data type — text, number, date, boolean, dropdown, etc.
  • Maximum length or range — for text fields, max characters; for numbers, min/max values
  • Mandatory or optional — can the form be submitted without this field?
  • Default value — is there a default? What is it?
  • Validation rules — "Must be a valid Bangladeshi NID format (17 digits)"; "Cannot be a future date"
  • Source — where does this data come from? (user input, API, calculated, imported from another module)
  • Linked to — what other tables or entities does this field reference?
Why it matters

Undefined data fields are the primary cause of post-launch data migration failures. When the new system launches and the old Excel data does not fit — fields too long, mandatory fields that were optional before, lookup tables that do not match — the cost of correction is always paid in production. A data dictionary prevents this.

Integration Requirements: APIs, Legacy Systems, ERP Connectors

Most new systems in Bangladesh's RMG and services sector do not exist in isolation. They integrate with: existing ERPs (Odoo, SAP, locally developed systems); third-party services (shipping tracking APIs, payment gateways, SMS/email services); and government systems (NBR VAT, BGMEA reporting portals).

Each integration requires its own specification:

  • What data flows in which direction — "Shipment status is pulled from the carrier API every 30 minutes and written to the shipment record"
  • Frequency — real-time, scheduled (every X minutes), or on-demand
  • Error handling — what happens if the external API is unavailable?
  • Authentication — API key, OAuth, basic auth
  • Data mapping — how do the external system's field names map to the internal data model?

Integration requirements are the area most likely to be underspecified by a BA and most likely to cause project overruns. Budget extra time here.

Non-functional Requirements: Performance, Security, Uptime

Non-functional requirements (NFRs) define the quality attributes of the system — how fast, how secure, how reliable, how usable. They are often stated as vague aspirations ("the system should be fast and secure") and need to be made as specific as functional requirements.

Performance

The system shall load the shipment dashboard within 3 seconds for up to 500 concurrent users. Report generation shall complete within 10 seconds for date ranges up to 12 months.

Security

All data transmission shall use TLS 1.2 or above. Passwords shall be stored using bcrypt with minimum 12-character length. Session shall expire after 30 minutes of inactivity.

Availability

The system shall maintain 99.5% uptime during business hours (8am–8pm BDT, Monday–Saturday). Planned maintenance windows shall be communicated 72 hours in advance.

Usability

A new merchandiser with no system training shall be able to create their first shipment record within 15 minutes of first login, using only in-system help text.

Scalability

The system architecture shall support a tenfold increase in transaction volume without code changes, through infrastructure scaling alone.

Data retention

Production orders shall be retained in the system for a minimum of 7 years to support buyer audit requirements. Deleted records shall be soft-deleted with an audit trail.

Bangladesh Compliance Context: What Specs Must Account For

Bangladesh's digital project requirements must account for a specific regulatory environment. Key compliance constraints to include in your NFRs:

  • NBR and VAT compliance — Any system handling invoicing, sales, or purchases must comply with Bangladesh VAT regulations and support the NBR's VAT return formats
  • BGMEA/BKMEA reporting — Factories in the garment sector may have reporting obligations to their industry association that affect what data the system must capture
  • Buyer audit trails — International buyers (UK, EU, US) increasingly require systems to produce documented audit trails for production records, worker records, and environmental compliance data
  • Data localisation — Bangladesh's Digital Security Act and emerging data protection frameworks may require that certain categories of data (especially worker personal data) be stored on servers within Bangladesh
  • Bangladesh Bank regulations — For systems handling foreign currency, remittance, or trade finance, Bangladesh Bank regulations may impose specific data fields and reporting requirements

Field Note

On a factory compliance system project, we discovered six months into development that the buyer required an audit trail for every change to a worker record — including who made the change, when, and what the previous value was. This is a standard requirement for SMETA audits, but it had never been mentioned in requirements because the local team did not know it was a system-level requirement (they assumed it was handled manually).

The database redesign to add full audit trail functionality took three additional weeks and delayed the launch. A document analysis of the buyer's compliance checklist at the start of the project would have surfaced this requirement in day one.

The Ambiguity Detector: Red-flag Phrases

Train yourself to hear these phrases as requirements failures in progress. When you encounter one, do not accept it — press for specificity.

"The system should be fast"

Fast relative to what? For whom? Under what load? Replace with: "The [function] shall complete within [X] seconds for [Y] concurrent users."

"Easy to use"

Usability is testable, but only with a specific benchmark. Replace with: "A user with no training shall be able to [complete task] within [X] minutes."

"And so on" / "etc."

These phrases in a requirement mean the BA gave up. "The report shall show PO number, buyer, status, etc." What is etc.? Name every field.

"Appropriate" / "adequate" / "sufficient"

These are judgment calls disguised as requirements. "The system shall provide adequate security" is not a requirement. Name the controls.

"The system should handle all cases"

Enumerate the cases. What are all cases? If you cannot list them, you do not know your own requirements.

"Standard industry practice"

Which standard? From which organisation? What specific practice? This phrase hides a requirements gap behind borrowed authority.

Template: SRS-lite Structure

Template

SRS-lite — document sections

1. Introduction — Project overview, document scope, definitions/acronyms, stakeholder sign-off record

2. Business Requirements — High-level goals the system must achieve (3–7 bullet statements)

3. Stakeholder Requirements — By role: what each user role needs the system to do for them

4. Functional Requirements — Numbered list, one requirement per statement, grouped by module/domain. Each formatted as: FR-[number]: The system shall [action] when [trigger], for [actor], producing [output].

5. Data Requirements — Entity list with field-level data dictionary (see section 3 format above)

6. Integration Requirements — Per integration point: direction, frequency, protocol, error handling

7. Non-functional Requirements — Performance, security, availability, usability, scalability, compliance

8. Out of Scope — Explicit list of what this project does NOT include. Sign-off required from sponsor.

9. Open Questions — Unresolved items with owner and due date

10. Change log — Version, date, author, summary of changes, approver

Reflection Questions

  1. Take a project you are currently working on and write three functional requirements in "The system shall…" format for the core module. Now check each one: is it testable? Does it name the actor? Does it specify the trigger and outcome?
  2. What compliance requirements apply to your current or most recent project that were not in the initial requirements document? How would document analysis at the start of the project have surfaced them?
  3. Audit one section of a requirements document you own or have access to. How many "red-flag phrases" from the Ambiguity Detector can you find? Rewrite three of them.
