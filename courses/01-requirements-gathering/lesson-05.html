Writing User Stories That Survive UAT

A user story that fails at UAT is not a UAT problem — it is a requirements problem that was deferred until the worst possible moment. Here is how to write stories that are precise enough to test, specific enough to build, and honest enough to reflect what the business actually needs.

The Anatomy of a User Story

The user story format was developed in the Agile community as a lightweight alternative to long-form requirements documents. Its power is not in the format itself — it is in what the format forces you to think about: who needs something, what they need, and why they need it.

The standard format: As a [role], I want [capability], so that [outcome/value].

Each of the three parts earns its place:

  • As a [role] — forces you to identify a specific user persona, not "the user" or "the system." Different roles have different needs and different privilege levels. Getting this wrong leads to features that serve the wrong person.
  • I want [capability] — describes the feature in terms of what the user can do, not how the system will implement it. "I want to see a dashboard" is a capability. "The system should use a bar chart" is a solution — and it belongs in design, not in requirements.
  • So that [outcome] — this is the most important and most frequently omitted part. The "so that" clause reveals the business value. It allows developers and testers to make judgment calls when implementation details are unclear, because they know what the feature is trying to achieve.

Worked Example — Shipment Tracking Module

As a merchandiser, I want to see the real-time status of each open shipment on a single screen, so that I can immediately identify delays and notify buyers before they contact us.

Notice what this story does not include: it does not specify the colour of the status indicators, the database structure, or how the data will be fetched. Those are implementation decisions. The story defines the need. Implementation is a conversation for the development team to have, informed by this story.

Acceptance Criteria: Given/When/Then

A user story without acceptance criteria is an intention, not a requirement. Acceptance criteria define exactly what "done" looks like — the conditions that a tester can verify, unambiguously, as met or not met.

The Given/When/Then (GWT) format, drawn from Behaviour-Driven Development (BDD), is the most useful structure for acceptance criteria:

  • Given — the starting state or context
  • When — the action the user takes
  • Then — the observable outcome

Acceptance Criteria — Shipment Status Story

AC1:

Given a merchandiser is logged in, When they open the shipments screen, Then they see a list of all open shipments with status (On track / At risk / Delayed) visible without scrolling.

AC2:

Given a shipment's expected date of arrival has passed without a confirmation, When the merchandiser views that shipment, Then its status shows as "Delayed" and is highlighted in red.

AC3:

Given a merchandiser clicks on a shipment, When the detail view opens, Then it shows: PO number, buyer name, origin port, destination port, carrier, container number, current status, and last update timestamp.

Three acceptance criteria is typically the right number for one user story. If you are writing seven or more, the story is too large and needs to be split. If you cannot write even one, the story is too vague to implement.

The UAT Test

For every acceptance criterion you write, ask yourself: "Could a tester who has never spoken to me verify whether this is met or not, using only this text?" If the answer is no — if it requires interpretation, memory of a conversation, or context that is not in writing — the criterion is not ready. Rewrite it until the answer is yes.

INVEST Criteria: The Quality Checklist

INVEST is an acronym that describes the properties of a well-formed user story. Run every story through this checklist before considering it ready for development.

I — Independent

Stories should be deliverable without being blocked by other stories. If story A can only be built after story B is complete, they should probably be a single story or an epic with a defined dependency. Highly coupled stories make sprint planning unreliable.

N — Negotiable

Stories are not contracts. The details can and should be refined through conversation between the BA, the developer, and the business. If a story is so rigid that no discussion is possible, it has become a specification, not a story.

V — Valuable

Every story should deliver value to a specific user or the business. Technical stories ("refactor the database schema") are valid but must be tied to a business value. "So that the system can handle 1,000 concurrent users during peak export season" is a value statement.

E — Estimable

Developers must be able to estimate the effort required. If they cannot — usually because the story is too vague, or requires technical investigation — that is a signal to either break it down or spike it (do a time-boxed investigation first).

S — Small

A story should be completable within one sprint (typically one to two weeks). If it cannot be, it is an epic — a container for multiple smaller stories. Break epics down until each story can be delivered and tested independently.

T — Testable

If you cannot write at least one acceptance criterion, the story is not ready. "The system should be user-friendly" is not testable. "The user can complete the shipment approval workflow in under three clicks" is testable.

Story Splitting Techniques

Epics and large stories need to be split into deliverable pieces. The most common splitting mistakes: splitting by technical component (front-end/back-end), which never delivers a complete feature; or splitting by timeline, which delivers half-finished functionality.

Effective splitting techniques:

  • By user workflow step — "Upload a file" and "Review uploaded file" are two stories, even though they are part of one workflow
  • By data type — "Track fabric shipments" and "Track accessories shipments" can be two stories if they have different rules
  • By business rule — One story for standard cases, a separate story for exception handling
  • By user role — "View shipment status (merchandiser)" and "Edit shipment status (logistics manager)" are different stories with different access rules
  • By acceptance criterion — If your story has seven ACs, split it into three stories, each with two or three ACs

The Anti-pattern Gallery

Too big

The story spans an entire module. "As an admin, I want to manage all user permissions" covers hundreds of scenarios.

❌ "manage all user permissions"
✓ "add a new user with role assignment"

Too vague

The story has no testable outcome. "The system should be easy to use" is a design aspiration, not a requirement.

❌ "so that it's easy to use"
✓ "so that I can complete approval in under 2 minutes"

Solution-first

The story describes implementation, not need. "As a user, I want a drop-down menu with 12 options" is a design decision disguised as a requirement.

❌ "I want a drop-down with 12 options"
✓ "I want to filter shipments by status"

Missing "so that"

Without the value clause, stories look trivial and developers cannot make judgment calls. The "so that" is what connects the story to the business.

❌ "I want to see shipment status"
✓ "…so that I can alert buyers before delays become complaints"

No acceptance criteria

Story is approved, sprint begins, development finishes — then at UAT nobody can agree what "done" looks like. The result is a three-day argument about scope.

❌ Story with 0 ACs in backlog
✓ Minimum 1 AC per story, ideally 2–3

Orphaned story

The story exists in a backlog but is not linked to any epics, test cases, or business requirements. When the feature fails UAT, nobody can trace it back to what was agreed.

❌ No parent epic, no test case link
✓ Linked to epic, to business req, to UAT test case

User Stories in RMG Context: Across the Three Tiers

In a garment factory, the same process — shipment tracking, for example — looks entirely different at three levels of the organisation. Write stories for each level separately, because their needs, their vocabulary, and their privilege requirements are distinct.

Three-tier Example — Shipment Tracking in RMG

Merchandiser level

As a merchandiser, I want to log container departure and arrival events against each open PO, so that the status timeline is accurate and buyer inquiries can be answered immediately.

Management level

As a production manager, I want to see a weekly summary of all shipments by buyer, status, and delay days, so that I can identify which buyers need proactive communication this week.

Executive level

As the MD, I want to see a single dashboard showing on-time shipment rate for the current quarter against the same period last year, so that I can assess operational performance in one view.

Reflection Questions

  1. Take a requirement from a project you know: "The system should allow users to manage their accounts." Rewrite it as a proper user story with at least two acceptance criteria in Given/When/Then format. Run it through the INVEST checklist. What needs to change?
  2. Find an example of the "solution-first" anti-pattern in a project you have worked on. What was the actual need behind the stated solution? How would a proper user story have captured it differently?
  3. For a garment factory's quality control system, write three separate user stories — one for a QC inspector, one for a QC supervisor, and one for the QC director. How do their needs and acceptance criteria differ?
